\section{Metodología Aplicada}%
\label{sec:methodology}

\begin{corregir}

Para estructurar las fases de desarrollo del proyecto y la gestión de los objetivos específicos se aplicó una
combinación de \textit{Scrum} con \textit{Crisp-DM}.
Se detallan a continuación ambas metodologías, junto con las técnicas y herramientas utilizadas.

\textit{Scrum} es un marco de trabajo que posibilita un proceso ágil y liviano para administrar y controlar el
desarrollo de \textit{software}.
Es iterativo e incremental y cada iteración o \textit{sprint} concluye con una pieza de \textit{software} ejecutable que
incorpora nueva funcionalidad.

En este caso, se debió aplicar ciertas modificaciones, ya que el equipo de trabajo estaba conformado por un único
integrante (también conocido como \textit{Single Scrum}), destacando que el autor del proyecto combinó los roles de
Desarrollador, \textit{Product Owner} y \textit{Scrum Master}.

En principio se planearon seis \textit{sprints} con una duración de dos semanas cada una, de forma que la totalidad del
proyecto contempla una duración aproximada de tres meses.
Una vez definidas las historias de usuario se procedió a dividirlas en tareas más pequeñas que terminaron
formando parte del \textit{backlog} del proyecto.
Al finalizar cada \textit{sprint} se compartió el avance hasta el momento con la tutora del proyecto para obtener
la retroalimentación correspondiente.
Si bien su aplicación abarca la totalidad de las tareas del proyecto, se destaca para las tareas del
desarrollo de \textit{software} que fueron necesarias en este proyecto como la implementación del \textit{endpoint} que,
a partir del contenido de un correo electrónico, extraiga la información relevante y también la implementación de una
interfaz web simple que muestre el itinerario agregado de un usuario.

Por su parte, \textit{Crisp-DM} es una metodología para el desarrollo de proyectos de \textit{data mining} (DM)
independiente de las herramientas de \textit{software}.
Esta metodología estructura el trabajo en seis fases: Comprensión del negocio, Análisis de los datos, Preparación de los
Datos, Modelado, Evaluación y Despliegue.
Su aplicación permitió en este proyecto guiar la creación de modelos de clasificación de los correos
electrónicos, la implementación de técnicas de extracción de información y la evaluación de los resultados obtenidos.

\end{corregir}

\subsection{Técnicas y Herramientas}%
\label{subsec:tools}

\begin{revisado}

En cuanto a las técnicas, en primer lugar se utilizaron tokenización y normalización, provenientes del área del
procesamiento de lenguaje natural (NLP). Para la clasificación de los correos electrónicos según el tipo de reserva que
contengan se utilizaron modelos de aprendizaje supervisado.
También se utilizó Named Entity Recognition para poder extraer la información relevante de cada mail de reserva, con
la ayuda de expresiones regulares para aquellos casos que respeten patrones predecibles.

Respecto al desarrollo de la solución se utilizaron diversas herramientas de acuerdo con los requerimientos de cada
componente.
Para el proceso de extracción, transformación y carga (ETL) se utilizó principalmente el lenguaje de programación
Java.
También se utilizó el lenguaje de programación R para los modelos de clasificación y extracción.
La visualización de los resultados se llevó a cabo utilizando Observable.
Para la implementación de la aplicación web que muestra el itinerario consolidado del usuario se utilizó Spring Boot.
Se evaluó además la posibilidad de incorporar un orquestador de flujos de trabajo como Apache Airflow.

Por otro lado, para la organización del proyecto en general se utilizaron varias funcionalidades del sitio GitHub.com
destacando el versionado del código fuente con \textit{Git} y el manejo del \textit{backlog} con GitHub Projects.
Se detallan a continuación ambas herramientas:

\end{revisado}

\subsection{Preparación de los Datos}%
\label{subsec:data-preparation}

En esta sección se mencionan los tipos distintos de mails que existen, el proceso de anonimización realizado para
que en el \textit{dataset} no aparezcan datos sensibles de los pasajeros que voluntariamente se prestaron a compartir
sus correos electrónicos y el uso de una herramienta para llevar un control de versiones del \textit{dataset}.

\subsubsection{Anonimización}

Los mails originales, con los cuales se conforma el dataset, contienen datos sensibles de los pasajeros como su nombre y
apellido, el número de ticket del vuelo, el número de afiliado a un programa de millas, los últimos dígitos de la
tarjeta de crédito con la que se realizó la compra, entre otros.

El primer paso en la anonimización consiste en eliminar todos los archivos adjuntos que contenga el correo electrónico.
Es común que las aerolíneas incluyan un PDF con un resumen de todos los datos de la reserva del vuelo y otro PDF con los
detalles de facturación correspondientes.

Para ello, se implementó un \textit{script} de Python (\texttt{remove\_sensitive.py}) que se encarga de analizar todos
los \texttt{.eml}\footnote{Un archivo \texttt{.eml} es un formato estándar de correo electrónico que almacena el mensaje
completo, incluyendo encabezados, cuerpo y archivos adjuntos, en un único archivo.} presentes en el directorio
\texttt{private\_inbox/} y dejar los nuevos \texttt{.eml} sin los adjuntos en el directorio \texttt{raw/}.

Luego de aplicar este \textit{script}, se realiza una búsqueda y reemplazo manual de todos los datos sensibles que
aparecen en el contenido del nuevo \texttt{.eml}.
Los datos se reemplazan por valores ficticios predefinidos:

\begin{itemize}
    \item Nombre: Martin Miles Young
    \item Mail: martin.young@sample.com
    \item Código de Reserva (PNR): Cualquier alfanumérico (por ejemplo ABC1DE)
    \item Ticket Number: 1234567890123
\end{itemize}

\subsubsection{DVC}

Data Version Control (DVC) existe como una alternativa a Git pero pensada para el versionado de \textit{datasets},
modelos y archivos grandes, complementando a Git sin sobrecargarlo.

Para esta implementación se creó un repositorio \texttt{dataset} dentro de la organización email2trip en DagsHub.
Se puede acceder desde https://dagshub.com/email2trip/dataset/.

Así para cada archivo de extensión \texttt{.eml} existe uno de extensión \texttt{.dvc}.
Este repositorio DVC está sincronizado con el repositorio Git \texttt{dataset} de GitHub, donde ambos se pueden apreciar
en la \autoref{fig:dataset-dvc-git-remote}, de forma que cada vez que se desee agregar un nuevo correo electrónico se
debe cargar en el repositorio Git el archivo de extensión \texttt{.dvc} y el archivo del correo electrónico
\texttt{.eml} debe cargarse únicamente en el repositorio DVC\@.

\subsection{Implementación del Modelo}%
\label{subsec:model-implementation}

En esta sección se detalla el proceso de implementación de la solución completa, incluyendo desde la lógica de
extracción de campos relevantes en un texto usando procesamiento de lenguaje natural (NLP) hasta el despliegue
de la aplicación web para que un usuario pueda registrarse, reenviar sus
correos electrónicos y obtener un itinerario de viaje integral.

\subsubsection{\textit{Pipeline} de extracción (\texttt{extractor-core})}

A continuación se detalla el \textit{pipeline} completo que inicia con un archivo de correo electrónico (de extensión
\texttt{.eml}) y que finaliza con la extracción de los datos relevantes del mismo.

Para el caso de correos electrónicos correspondientes a vuelos se busca obtener el código de la reserva, el nombre del
pasajero y el detalle de uno o más segmentos de vuelo.
Cada segmento se compone de un código de vuelo, el aeropuerto de origen, el aeropuerto de destino y la fecha y hora de
inicio de vuelo.
\begin{trenes}
En cuanto a los correos correspondientes a viajes en tren se espera extraer la misma información, donde cada segmento
tendrá un número de tren, la estación de origen, la estación de destino y la fecha y hora de inicio del viaje.
\end{trenes}

\subparagraph{Preprocesamiento para códigos de vuelo}

Otro preprocesamiento adicional que se aplica es la búsqueda mediante expresiones regulares de cadenas que puedan
corresponder con códigos de vuelo (Ver \autoref{lst:normalize-flight-numbers}).
Un código de vuelo~\cite{iata-airline-codes} consiste en dos letras, seguido de un número
de entre 2 y 4 cifras.
Por ejemplo \texttt{AC8633} corresponde al vuelo 8633 de la aerolínea Air Canada.
En caso de que el mail contenga cadenas como \texttt{AC-8633} o \texttt{AC 8633} este preprocesamiento elimina el guion
o el espacio para que siempre se obtenga la cadena \texttt{AC8633}, facilitando así la tarea de la próxima etapa del
\textit{pipeline} para identificar códigos de vuelo usando una única convención.

\begin{lstlisting}[
    style=javacodestyle,
    caption={Expresiones regulares utilizadas para la normalización de códigos de vuelo.},

  Pattern FLIGHT_NUMBER_WITH_HYPHEN_SEPARATOR =
    Pattern.compile("\\b([A-Z][A-Z0-9]|[0-9][A-Z]|[A-Z][0-9])\\s*[-–]\\s*(\\d{2,4})\\b");
  Pattern FLIGHT_NUMBER_WITH_SPACE_SEPARATOR =
    Pattern.compile("\\b([A-Z][A-Z0-9]|[0-9][A-Z]|[A-Z][0-9])\\s+(\\d{2,4})\\b");

\paragraph{Stanford CoreNLP}

Una vez que se cuenta con una instancia de \textit{String} con únicamente el contenido a simple vista de un correo
electrónico se envía este texto al \textit{pipeline} de~\cite{stanford-corenlp}.
Stanford CoreNLP es una biblioteca de procesamiento de lenguaje natural desarrollada por Stanford que permite analizar
texto automáticamente, por ejemplo detectando tokens, lemas, categorías gramaticales, entidades y relaciones, entre
otros.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=1.00\textwidth]{images/pipeline}
    \figcaption{Diagrama del \textit{pipeline} de CoreNLP}{\sourcelink{CoreNLP}{https://stanfordnlp.github.io/CoreNLP/}}%
    \label{fig:corenlp-pipeline}
\end{figure}

Como se puede apreciar en la \autoref{fig:corenlp-pipeline}, el mismo cuenta con múltiples etapas que detallamos a
continuación:

\FloatBarrier

\subparagraph{Tokenization}

La tokenización consiste en convertir texto en tokens.
La forma más elemental consiste en dividir cada vez que aparece un espacio.
Por ejemplo para el texto ``Thanks for flying.'' la tokenización genera cuatro tokens:% chktex 38
``Thanks'', ``for'', ``flying'' y ``.''.% chktex 38
CoreNLP realiza la separación del texto en \textit{tokens} usando una colección de reglas ya previstas.
En el archivo de configuración se indica la propiedad \texttt{tokenize.language=en} para que utilice las reglas del
idioma inglés.
Desde este momento la implementación estará limitada exclusivamente a correos electrónicos de reservas redactados en
inglés.

\subparagraph{Sentence Splitting}

La separación en oraciones consiste en convertir un texto en oraciones.
Por ejemplo ``Thanks for flying. Fly safe.''.% chktex 38
será separado en las oraciones ``Thanks for flying.'' y ``Fly safe.''.% chktex 38
Aquí también CoreNLP realiza la separación en oraciones usando una colección de reglas ya previstas.

\subparagraph{Parts of Speech}

Partes del discurso asignan \textit{tags} o etiquetas a los \textit{tokens} para marcarlos según su categoría
gramatical.
Por ejemplo para el texto ``Marie was born in Paris.'' se etiqueta a ``Marie'' como \texttt{NNP}% chktex 38
(nombre propio singular), ``was'' como \texttt{VBD} (verbo pasado), ``born'' como \texttt{VBN} (verbo participio pasado),
``in'' como \texttt{IN} (preposición), ``París'' nuevamente como \texttt{NNP} y el punto final ``.'' como signo de% chktex 38
puntuación.

\subparagraph{Lemmatization}

Aquí se asocia una palabra con su lema o forma de diccionario.
Por ejemplo para la palabra ``was'' se mapea por la palabra ``be''.
Este paso lleva todo el contenido a un solo tiempo verbal para simplificar la tarea de las próximas etapas.

\subparagraph{Named Entity Recognition}

En esta etapa, la más compleja de todo el pipeline de NLP, se reconocen entidades en el texto como personas o nombres
de empresas.
La librería utiliza uno o más modelos de secuencia de \textit{machine learning} para etiquetar las entidades.
Además permite la ejecución de reglas específicas para etiquetar, por ejemplo, fechas, tiempos y valores numéricos.

Para el idioma inglés la biblioteca reconoce por defecto 24 tipos distintos de entidades:
\begin{itemize}
    \item Nombres reconocidos: \texttt{PERSON}, \texttt{LOCATION}, \texttt{ORGANIZATION}, \texttt{MISC}
    \item Numéricos: \texttt{MONEY}, \texttt{NUMBER}, \texttt{ORDINAL}, \texttt{PERCENT}
    \item Temporales: \texttt{DATE}, \texttt{TIME}, \texttt{DURATION}, \texttt{SET}
    \item Otros: \texttt{EMAIL}, \texttt{URL}, \texttt{CITY}, \texttt{STATEORPROVINCE}, \texttt{COUNTRY},
    \texttt{NATIONALITY}, \texttt{RELIGION}, \texttt{TITLE}, \texttt{IDEOLOGY}, \texttt{CRIMINALCHARGE},
    \texttt{CAUSEOFDEATH}, \texttt{HANDLE}
\end{itemize}

A simple vista se puede apreciar que no existe una entidad aeropuerto o aerolínea por lo que es necesario crear reglas
adicionales para facilitar el reconocimiento de nuevos tipos de entidades.

\subsubparagraph{Aeropuertos}

Un aeropuerto de pasajeros se identifica por un código de tres letras denominado~\cite{iata-airport-code}.
A excepción de aeropuertos muy pequeños, cualquier aeropuerto cuenta con un identificador único de tres letras emitido
por IATA\@.

Para esta implementación se utilizó el \textit{dataset} público de~\cite{ourairports-data} que ofrece
la descarga de un archivo CSV con más de 84.000 registros que
representan a todos los aeropuertos del mundo.
Se procede a filtrar el mismo para considerar únicamente los aeropuertos que tengan un código IATA (descartando
aeropuertos pequeños).
Por ejemplo el aeropuerto ``Montreal / Pierre Elliott Trudeau International Airport'' se identifica con el código IATA
``YUL'' (Ver \autoref{lst:airports-csv-line}).

\begin{lstlisting}[
    style=basecodestyle,
    caption={Registro de ejemplo del \textit{dataset} Our Airports Data.},
    label={lst:airports-csv-line}
]

1928,CYUL,large_airport,Montreal / Pierre Elliott Trudeau International Airport,45.467837,-73.742294,118,NA,CA,CA-QC,Montréal,yes,CYUL,YUL,CYUL,YUL,http://www.admtl.com/passager/Home.aspx,https://en.wikipedia.org/wiki/Montréal–Trudeau_International_Airport,"YMQ, Dorval Airport"
\end{lstlisting}

Para cada aeropuerto con código IATA se procede a crear un conjunto de reglas para que el \textit{pipeline} pueda
reconocer con éxito la entidad \texttt{AIRPORT\_CODE} mencionando todos los códigos de aeropuerto existentes.
Se creó un archivo \texttt{airport.rules} que asocia tanto el código IATA como la combinación de código IATA seguido del
nombre de la ciudad donde se ubica el aeropuerto.
Retomando el ejemplo de Montreal, el contenido del archivo de reglas es el siguiente
(Ver \autoref{lst:airport-rules-yul}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={Reglas de \texttt{airport.rules} para el aeropuerto YUL.},
    label={lst:airport-rules-yul}
]
YUL	AIRPORT_CODE	O,MISC,ORGANIZATION	2.0
YUL Montréal	AIRPORT_CODE	O,MISC,ORGANIZATION	2.0
\end{lstlisting}

La estructura de los archivos de reglas consiste en hasta 4 columnas separadas por un \textit{tab}.
En la primer columna se indica la cadena a corresponder, luego se indica  el tipo de categoría a asociar.
En la tercer columna se indican las categorías reconocidas que pueden ser reemplazadas (en el ejemplo estamos indicando
que si fue reconocido como \texttt{MISC} o como \texttt{ORGANIZATION} entonces es más importante \texttt{AIRPORT\_CODE}
y por lo tanto debería reconocerlo así).
Por último con el valor \texttt{2.0} estamos indicando la prioridad, que resulta útil para el desempate cuando dos
reglas aplican para una misma cadena.
De esta forma CoreNLP entiende que frente al tag ``YUL'' lo debe reconocer como un nuevo tipo \texttt{AIRPORT\_CODE}.

Debido a que CoreNLP puede reconocer a una cadena ``YUL Montréal'' como una única entidad, y anotarla como
\texttt{ORGANIZATION}, entonces necesitamos de una segunda regla para cada aeropuerto donde se ubica primero el código
IATA y luego el nombre de la ciudad correspondiente.

La particularidad de los nombres de los aeropuertos es que el modelo puede fácilmente con el nombre de una persona,
por ejemplo del pasajero del vuelo.
Siguiendo con el mismo ejemplo, para la cadena ``Charles de Gaulle Airport'' el modelo lo anota como \texttt{PERSON}.
Es por eso que para cada aeropuerto se crean hasta tres reglas adicionales indicadas en el archivo
\texttt{airport-names.rules} donde para un nombre proveniente del dataset como ``Montreal / Pierre Elliott Trudeau
International Airport'' se incluye no sólo la cadena exacta, sino también la misma sin la palabra ``Airport'' y otra sin
la cadena ``International Airport''.
De esta forma, para el aeropuerto en cuestión, se obtienen las siguientes reglas
(Ver \autoref{lst:airport-names-rules-yul}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={Reglas de \texttt{airport-names.rules} para el aeropuerto YUL.},
    label={lst:airport-names-rules-yul}
]
Montreal / Pierre Elliott Trudeau	AIRPORT_NAME	O,MISC,ORGANIZATION,PERSON	2.0
Montreal / Pierre Elliott Trudeau International	AIRPORT_NAME	O,MISC,ORGANIZATION,PERSON	2.0
Montreal / Pierre Elliott Trudeau International Airport	AIRPORT_NAME	O,MISC,ORGANIZATION,PERSON	2.0
\end{lstlisting}

A diferencia del fragmento de reglas anterior aquí es necesario precisar que el tipo de entidad \texttt{PERSON} es
reemplazable por una nueva entidad \texttt{AIRPORT\_NAME}.
En principio para el procesamiento de los correos de vuelos no es necesario extraer el nombre del aeropuerto ya que los
identificadores IATA anotados como \texttt{AIRPORT\_CODE} son únicos.
De todas formas la identificación es útil para que el nombre del aeropuerto no sea identificado como \texttt{PERSON} y
así se facilita la tarea de identificar el nombre del pasajero de la reserva.

Por último se creó una regla de~\cite{stanford-tokensregex} donde se
rescata al aeropuerto cuando el texto visible queda con un formato como ``EZE Buenos Aires Departing at'' o ``EZE Buenos
Aires Arriving at:''. % chktex 38
Para ello, en el archivo \nolinkurl{contextual-airport.tokensregex.rules} se definió lo siguiente
(Ver \autoref{lst:contextual-airport-tokensregex}):

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Regla contextual para identificar el código del aeropuerto.},
    label={lst:contextual-airport-tokensregex}
]
ner = { type: "CLASS", value: "edu.stanford.nlp.ling.CoreAnnotations$NamedEntityTagAnnotation" }
{
  ruleType: "tokens",
  pattern: ((([{word:/[A-Z]{3}/}] [{word:/[A-Z][A-Za-z'.-]*/}] [{word:/[A-Z][A-Za-z'.-]*/}]?)) [{word:/Departing|Arriving/}] [{word:/at/}] [{word:/:/}]?),
  action: Annotate($2, ner, "AIRPORT_CODE"),
  result: "AIRPORT_CODE"
}
\end{lstlisting}

Como se puede apreciar se define un patrón donde primero aparece el código IATA de tres letras, seguido de 0 o más
palabras y luego las palabras exactas ``Departing'' o ``Arriving''.
Esta regla permite identificar al aeropuerto según el contexto en el que se ubica, pudiendo anotar únicamente el texto
correspondiente al aeropuerto como \texttt{AIRPORT\_CODE} y descartando el resto (en la parte de \texttt{action} se
indica \texttt{Annotate(\$2,\ldots)} para anotar el segundo grupo capturado, esto es, ``EZE Buenos Aires'').% chktex 36

A pesar de contar con reglas para la totalidad de códigos de aeropuertos existentes esto trae aparejado el problema de
los falsos positivos.
Una sigla en inglés como ``MAY'' que corresponde a un \textit{modal verb} será identificado incorrectamente como el
``Clarence A. Bain Airport'' en las Bahamas.
Del mismo modo la cadena ``MAX'' utilizada para indicar el máximo peso aceptado por la aerolínea en el equipaje ahora
será identificado como el ``Ouro Sogui Airport'' ubicado en Senegal.
Para estos casos extremos identificados, que siempre corresponden a aeropuertos pequeños, se decidió utilizar la
propiedad \texttt{ner.additional.regexner.commonWords} para indicarle a CoreNLP las palabras que deben ser tomadas como
palabras comunes del idioma inglés y así excluirlas de cualquier identificación.
Las palabras definidas en la implementación son las siguientes: ``API'', ``GET'', ``GST'', ``HST'', ``MAX'', ``MAY'', y
``VOL''.

\subsubparagraph{Códigos de Vuelos y Aerolíneas}

Anteriormente se comentó el preprocesamiento que se realiza antes de iniciar el pipeline de CoreNLP y que involucra a
los códigos de vuelo definidos por IATA\@.
Ahora es momento de obtener el token ``AC8633'' y reconocerlo como una entidad de tipo \texttt{FLIGHT\_NUMBER}.
Para ello se utilizó el \textit{dataset} público de aerolíneas de~\cite{openflights-data}
que ofrece un CSV con más de 6.000 registros que representan a todas las aerolíneas del mundo.
Por ejemplo (Ver \autoref{lst:airlines-csv-line}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={Registro de ejemplo del \textit{dataset} OpenFlights.},
    label={lst:airlines-csv-line}
]
C1,C2,C3,C4,C5,C6,C7,C8
330,Air Canada,\N,AC,ACA,AIR CANADA,Canada,Y
\end{lstlisting}

Para cada aerolínea se cuenta con el código de dos letras de IATA (``AC''), el nombre y el país de origen de la misma,
entre otros.
Con esta información se puede crear un nuevo conjunto de reglas que modele el comportamiento de encontrar todos los
códigos de aerolínea (compuesto por dos caracteres alfanuméricos) seguidos de entre 2 y 4 dígitos que corresponden al
número de vuelo.
Para la aerolínea en cuestión el archivo de reglas es el siguiente (Ver \autoref{lst:airlines-rules-air-canada}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={Regla de \texttt{airlines.rules} para Air Canada.},
    label={lst:airlines-rules-air-canada}
]
AC\d{2,4}	FLIGHT_NUMBER	O,MISC	2.0
\end{lstlisting}

donde el nuevo tipo de entidad \texttt{FLIGHT\_NUMBER} tiene prioridad por sobre una entidad de tipo \texttt{MISC}.
Este conjunto se reglas se ubica en el archivo \texttt{airlines.rules}.

Utilizando el mismo \textit{dataset} se crea otro conjunto de reglas, ubicado en el archivo \texttt{airline-names.rules}
para definir el nuevo tipo de entidad \texttt{AIRLINE\_NAME} y que tenga prioridad por sobre tipos \texttt{MISC} y
\texttt{ORGANIZATION} (Ver \autoref{lst:airline-names-rules-air-canada}).

\begin{lstlisting}[
    style=basecodestyle,
    caption={Regla de \texttt{airline-names.rules} para Air Canada.},
    label={lst:airline-names-rules-air-canada}
]
AIR CANADA	AIRLINE_NAME	O,MISC,ORGANIZATION	2.0
\end{lstlisting}

Nuevamente, si bien no es necesario extraer el nombre de la aerolínea (ya que la primera porción del código de vuelo
es un identificador único), el hecho de anotar un nombre como nombre de aerolínea ayuda a que existan menos partes del
contenido sin anotar y que podrían ser candidatas a identificaciones incorrectas.

\subsubparagraph{Código de Reserva (PNR)}

Otro campo relevante para la extracción es el código de reserva.
Aquí también IATA define el \texttt{PNR} o~\cite{passenger-name-record} y consiste en 6
caracteres alfanuméricos.
Para sumar dificultad de la extracción se recuerda que el código de vuelo (como ``AC8633'') ya está cumpliendo con el
formato de \texttt{PNR}.
Es por eso que aquí la estrategia para la extracción es distinta y consiste en identificar el \texttt{PNR} por contexto.
Se establecen un conjunto de reglas \texttt{TokensRegex} para extraer el \texttt{PNR} cuando aparece luego de frases como
``Confirmation Number'', ``Booking Code'', ``Booking Reference'', entre otras.
En el archivo \texttt{pnr.tokensregex.rules} se indican seis reglas.
El principio del archivo es el siguiente (Ver \autoref{lst:pnr-tokensregex-rules}):

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Una de las reglas para identificar el código de reserva del vuelo.},
    label={lst:pnr-tokensregex-rules}
]
ner = { type: "CLASS", value: "edu.stanford.nlp.ling.CoreAnnotations$NamedEntityTagAnnotation" }
$PNR_CODE = "/[A-Z0-9]{6}/"
$SEPARATOR = "/[:-]/"
{
  ruleType: "tokens",
  pattern: ([{word:/[Bb]ooking/}] [{word:/[Rr]eference/}] [{word:$SEPARATOR}]? ([{word:$PNR_CODE}])),
  action: Annotate($1, ner, "PNR_CODE"),
  result: "PNR_CODE"
}
\end{lstlisting}

Con estas reglas se define un nuevo tipo de entidad \texttt{PNR\_CODE} listo para su extracción.

\subsubparagraph{SUTime: Fechas y horas}

En un mismo mail de confirmación de reserva pueden aparecer múltiples fechas y horas correspondientes por ejemplo al
despegue o aterrizaje de cualquiera de los tramos de la reserva.
Además es común que se indique la fecha en que se realizó la confirmación o incluso la fecha límite hasta donde se
pueden realizar modificaciones a la reserva sin costo adicional.
Es por eso que en esta etapa de anotación sólo resulta útil anotar todos los campos correspondientes a fecha y hora como
\texttt{TIME} y la extracción con contexto se realizará en la siguiente etapa.
Solo así podremos asociar por ejemplo la fecha y hora de despegue al primer tramo de una reserva y no al segundo.

Las reglas por defecto fueron suficientes para la mayoría de los correos electrónicos, ya que CoreNLP provee la
biblioteca~\cite{stanford-sutime} que permite reconocer por ejemplo fechas con
múltiples formatos como ``31-12-2026'', ``31/12/2026'' o ``2026-31-12''.

Sólo fue necesaria una regla \texttt{TokensRegex} extra (Ver \autoref{lst:sutime-boarding-pass-rule}):

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Regla para identificar fechas en \textit{boarding passes}.},
    label={lst:sutime-boarding-pass-rule}
]
ner = { type: "CLASS", value: "edu.stanford.nlp.ling.CoreAnnotations$NamedEntityTagAnnotation" }
{
  ruleType: "tokens",

\subsubparagraph{Extracción de información para trenes}

El reconocimiento de información en el ambiente de trenes plantea ciertos desafíos cuando se lo compara con el de
vuelos.
El hecho de que no exista una organización como IATA que estandarice la información hace no sólo que existan diferencias
respecto a los vuelos sino que plantea variantes entre redes de trenes distintas.

Para esta implementación se contemplan dos redes de trenes: la canadiense y la europea.
A simple vista ya existen diferencias de formato y longitud en los campos relevantes a extraer.

Tal como se aprecia en la \autoref{fig:train-booking} el tren canadiense es el tren 33 (indicado como Train 33) mientras
que en Europa, donde es habitual que varias compañías de trenes distintas operen sus servicios en una misma estación,
el servicio de DB se identifica como RJX 260, acercándose al concepto de código de vuelo ya mencionado anteriormente.
Se plantean entonces un conjunto de reglas para poder extraer el identificador de tren, contemplando que puede ser sólo
un número, como así también una cadena alfanumérica que puede incluir un espacio.
Al mismo tiempo se descarta la extracción de la empresa de tren operadora del viaje debido a las diferencias planteadas.

Además del identificador del tren, otro campo que plantea matices dependiendo la empresa o la red de trenes es el código
de reserva.
En la misma figura se ve que para la empresa canadiense el código de reserva es un alfanumérico de seis caracteres
(coincidente con el concepto de PNR establecido por la IATA) mientras que para el sistema europeo el concepto de un
identificador de la reserva no es relevante y en el correo se indica un \textit{order number} que consiste en un número
de doce cifras.
Por lo tanto se plantean reglas para poder extraer ambos tipos de código de reserva.

La detección de segmentos de tren sí plantea similitudes con la de vuelos ya que conceptualmente coinciden.
En lugar de un aeropuerto de origen y destino se deben buscar las estaciones de tren de origen y destino.
La fecha y hora de salida y el nombre del pasajero son campos que no agregaron diferencias a la lógica de extracción.
Sí corresponde detenerse es en la identificación de estaciones de tren ya que se necesita recurrir a
\textit{datasets} distintos.

\subsubparagraph{Estaciones de Tren}
\label{subsubpar:train-stations}

Otro efecto de que no exista una organización que regule todos los viajes en tren del mundo hace que sea más difícil
conseguir un \textit{dataset} que contenga, por ejemplo, todas las estaciones de tren del mundo.
En busca de estandarizar la fuente de información, independientemente del país o sistema de trenes, se optó por el
formato GTFS\footnote{\textit{General Transit Feed Specification}} que consiste en un formato de datos estandarizado
para que los operadores de transporte puedan definir los detalles de sus servicios (estaciones, rutas, horarios, etc.).
Además del GTFS estático existe un GTFS en tiempo real que ofrece, por ejemplo, la ubicación de los trenes en el mapa.

Del portal MobilityDatabase~\cite{mobilitydatabase} se descargan los archivos GTFS del sistema
canadiense, donde el único operador es VIA Rail y cuenta con 168 estaciones de tren de Canadá y Estados Unidos.

Para el sistema europeo también se recurre al mismo portal para descargar los archivos GTFS de 14 operadores
(entre ellos DB de Alemania, ÖBB de Austria y SBB de Suiza).
El \textit{dataset} cubre 420 estaciones de tren de 17 países europeos.

De los archivos obligatorios que el estándar GTFS requiere sólo se utiliza el archivo \texttt{stops.txt} donde cada
registro corresponde a una estación.
La información es limitada, ya que si bien se incluye el nombre de la estación y sus coordenadas, no se incluye la
ciudad ni el país donde se ubica la estación.

Un desafío que se circunscribe al sistema de trenes europeo es la detección del nombre de una estación de tren en
distintos idiomas y con distintos sufijos.
Si bien la implementación actual está limitada a correos electrónicos en inglés, las empresas de tren suelen incluir
no solo el nombre original de la estación sino también una variante en inglés o en el idioma de la compañía de tren.

Es por eso que es necesario recurrir a un \textit{dataset} adicional proveniente de la empresa
\sourcelink{Trainline EU}{https://github.com/trainline-eu/stations} que provee mucha más información para cada estación
de tren.
En primer lugar indica el identificador que coincide con la especificación GTFS\@.
Esto permite agregar información adicional directamente a cada estación del \textit{dataset} original.
Los campos que se destacan son los de internacionalización, como las columnas \texttt{info:de}, \texttt{info:en} y
\texttt{info:cs} donde para cada estación se indica el nombre en alemán, inglés y checo respectivamente.
Por ejemplo para la estación austríaca Wien Hbf, en \texttt{info:en} se obtiene Vienna y en \texttt{info:cs} Vídeň.
Para el caso de la estación Praha hl.n., en \texttt{info:de} se obtiene Prag y en \texttt{info:en} se obtiene Prague.

Por último es necesario contemplar un tratamiento particular para los sufijos que existen en los nombres de las
estaciones, donde aquí también varía según el idioma.
En el idioma checo el sufijo de la estación principal es la abreviatura ``hl.n.''.
Entonces para la estación ``Praha hl.n.'' es necesario reconocer también las cadenas ``Praha, hl.n.'', ``Prague'',

\paragraph{Extracción de información}

La salida del \textit{pipeline} de CoreNLP implementado en \nolinkurl{xyz.email2trip.pipeline.FlightPipeline} consiste en
una instancia de \nolinkurl{edu.stanford.nlp.pipeline.CoreDocument} que cuenta con un conjunto de entidades anotadas.
A continuación se indica cómo se realiza la extracción de los campos relevantes a partir de la búsqueda de entidades de
un tipo en particular.

\subsubparagraph{Extracción de campos únicos: PNR y Pasajero}

Para el código de reserva (PNR) se busca una entidad de tipo \texttt{PNR\_CODE}.
En caso de que exista más de una se obtiene la que apareció primero.
En cuanto al nombre del pasajero primero se buscan las entidades de tipo \texttt{PERSON}.
Como aquí es común encontrar más de una se deben ponderar todas las entidades bajo un criterio determinístico y así
elegir la que tiene una mejor puntuación, esto es, la que es más probable que represente el nombre del pasajero.
En las pruebas, y retomando el ejemplo de la palabra ``Max'' la misma era etiquetada como \texttt{PERSON} con una
\textit{confidence} mayor que, por ejemplo, el nombre verdadero del pasajero del correo (``Martin Miles Young'').
CoreNLP ofrece el método \texttt{edu.stanford.nlp.pipeline.CoreEntityMention\#entityTypeConfidences} que permite obtener
el valor de \textit{confidence} de una entidad candidata para el tipo deseado.
Por ejemplo una entidad puede tener un valor de \textit{confidence} para \texttt{PERSON} de 0.9 y al mismo tiempo uno de
0.75 para el tipo \texttt{ORGANIZATION}.
Es por eso que el proceso para conseguir el nombre del pasajero consiste en obtener los valores de \textit{confidence}
de todas las entidades para el tipo \texttt{PERSON} y quedarse con el de mayor valor.
El criterio anterior no fue necesario y se necesitó realizar un filtro extra para darle mayor prioridad a entidades que
más se parecieran a nombres completos.
Contando la cantidad de palabras de una entidad se prioriza aquella que tenga al menos dos palabras, para corresponder
con el patrón de nombre y apellido.
Así una cadena como ``Max'' no luce como un nombre y apellido pero ``Martin Miles Young'' sí.

\subparagraph{Extracción de campos múltiples: Segmentos de vuelo}

La extracción de los datos del vuelo requiere cierta lógica adicional dado que se espera extraer más de un campo del
mismo tipo de entidad.
Por ejemplo para una reserva con un solo vuelo se necesita conocer el aeropuerto de origen o despegue y el aeropuerto de
destino o aterrizaje.
Allí ya se necesitan dos entidades de tipo \texttt{AIRPORT\_CODE} donde, por la naturalidad del idioma inglés, la primer
entidad corresponderá al aeropuerto de origen y la segunda al de destino.
También se busca una entidad de tipo \texttt{FLIGHT\_NUMBER} para el código del vuelo en cuestión.
Respecto a la fecha y hora de origen del vuelo, se necesita información de contexto ya que la primer entidad de tipo
\texttt{DATE} que aparezca en el correo podría representar información distinta.
Es necesario que la entidad de tipo \texttt{DATE} esté cerca del código del aeropuerto destino.
Lo mismo aplica para la hora de despegue.
La lógica mencionada para aeropuerto origen, destino, fecha y hora de origen y código de vuelo puede encapsularse en lo
que se denomina la identificación de un tramo de vuelo.
En un único mail de reserva de un vuelo se pueden encontrar uno o más segmentos o tramos.
Por lo tanto se procederá a identificar tramos donde cada tramo se compone de dos entidades de tipo
\texttt{AIRPORT\_CODE}, un \texttt{DATE} y un \texttt{TIME} para el origen y un \texttt{FLIGHT\_NUMBER}.
Para ello se implementaron cuatro \textit{layouts} posibles, es decir, cuatro formas en que podrían aparecer estas
entidades, contemplando así la totalidad de los correos de reservas presentes en el \textit{dataset}.

\begin{itemize}
    \item Opción 1: El patrón clásico que utiliza la mayoría de las aerolíneas
    \begin{itemize}
        \item Regla:
        \texttt{AIRPORT \ldots\ TIME \ldots\ AIRPORT\_CODE}\\
        \texttt{\ldots\ TIME \ldots\ FLIGHT\_NUMBER}
        \item Ejemplo: ``LGA 08:35 YUL 09:59 AC8633''
    \end{itemize}
    \item Opción 2: Otra forma bastante utilizada donde primero aparece el código del vuelo
    \begin{itemize}
        \item Regla:
        \texttt{FLIGHT\_NUMBER \ldots\ AIRPORT\_CODE \ldots\ TIME}\\
        \texttt{\ldots\ AIRPORT\_CODE \ldots\ TIME}
        \item Ejemplo: ``AC132 YVR Vancouver Departing at 13:00 YYZ Toronto Arriving at 20:28''
    \end{itemize}
    \item Opción 3: Un layout partido donde las fechas y horas aparecen al final
    \begin{itemize}
        \item Regla:
        \texttt{AIRPORT\_CODE \ldots\ AIRPORT\_CODE \ldots\ FLIGHT\_NUMBER}\\
        \texttt{\ldots\ DATE TIME \ldots\ DATE TIME}
        \item Ejemplo: ``EZE ATL DL110 Departs 07/20/2020 8:20 PM Arrives 07/21/2020 5:48 AM''
    \end{itemize}
    \item Opción 4: El más utilizado por los correos de tipo boarding pass donde primero aparece la fecha del embarque
    \begin{itemize}
        \item Regla:
        \texttt{DATE \ldots\ FLIGHT\_NUMBER \ldots\ AIRPORT\_CODE \ldots\ TIME}\\
        \texttt{\ldots\ AIRPORT\_CODE \ldots\ TIME}
        \item Ejemplo: ``22SEP25 OS208 FRA 12:50 VIE 14:15''
    \end{itemize}
\end{itemize}

En todas ellas la notación \texttt{\ldots} acepta cero o más entidades de cualquier tipo distinto a los buscados.
El último paso es evitar conservar dos segmentos que resultan equivalentes, esto es, que tienen los mismos datos.
Esto es común en confirmaciones de vuelos donde la información aparece primero con los datos del pasajero y luego
aparece por segunda vez sólo para indicar la política de equipaje para cada segmento.
Una vez que se extrajeron los campos relevantes de una reserva se envía esa información al proceso de enriquecimiento o
\textit{enrichment} para conseguir más información a partir de los identificadores únicos que resultan algunas de las
entidades identificadas.

\subsubsection{Recomendación de POIs \texttt{extractor-recommender}}

Esta sección cubre la funcionalidad adicional de ofrecer recomendaciones de puntos de interés que el usuario podría
visitar en cada una de las ciudades destino de la reserva (ciudades según los aeropuertos destino de cada segmento).

En lugar de utilizar una base de datos de POIs (\textit{points of interest}) se optó por utilizar un
\textit{Large Language Model} (LLM) de forma local al que se le solicitan recomendaciones de POIs para un destino y se
exige que la respuesta del modelo sea estructurada en formato JSON\@.

\paragraph{Intereses del Usuario}

Las recomendaciones sobre lugares a visitar dependen de las preferencias o intereses de cada persona.
Algunas pueden preferir lugares más relacionados a la naturaleza mientras que a otros les interesen más las compras y
elijan visitar tiendas o mercados.
Es por eso que se definió una forma de categorizar a los usuarios en función de los intereses de una forma simple.
Se definen cinco categorías de intereses:

\begin{itemize}
    \item \texttt{CULTURE\_HISTORIC}: Museos o centros históricos
    \item \texttt{NATURE}: Parques Nacionales, plazas, lagos
    \item \texttt{SHOPPING}: Tiendas y mercados
    \item \texttt{NIGHTLIFE}: Vida nocturna
    \item \texttt{RELIGIOUS}: Iglesias y conventos.
\end{itemize}

El usuario al momento de solicitar una recomendación puede elegir hasta dos de estos intereses.
El proceso de recomendación deberá priorizar los puntos de interés que se ajustan a los intereses del usuario, sin
impedir que pudieran recomendarse también algunos puntos de interés más relacionados a intereses no seleccionados.

\paragraph{\textit{Stack} LLM: Gemma 4 + Ollama + Spring AI}% chktex 13
\label{par:stack-llm}

Para esta implementación se decidió utilizar Gemma 4 E4B como LLM\@.
Es un modelo recientemente lanzado que pertenece a la familia de modelos abiertos desarrollados por Google DeepMind.
Ofrece compatibilidad con más de 140 idiomas y la versión E4B cuenta con 4.500 millones de parámetros totales efectivos.
De acuerdo a lo enunciado en la tarjeta del modelo, Gemma 4 se recomienda para tareas como la generación de texto, la
programación y el razonamiento.
La variedad de modelos (en función de la cantidad de parámetros) permite que estos puedan ser implementados en entornos
que van desde teléfonos de alta gama hasta laptops y servidores, es decir, sin necesidad de contar con una \textit{GPU}
potente.

El modelo se ejecuta en Ollama, una herramienta que permite ejecutar modelos localmente, tanto en CPU como en GPU,
exponiendo una \textit{API HTTP} para interactuar con el modelo y así facilitar la integración en aplicaciones.
En la implementación se utiliza Ollama dentro de un contenedor de Docker por lo que el modelo se ejecuta sólo en CPU\@.
Sin embargo el uso de Ollama permite que, al momento de realizar el \textit{deploy} de la aplicación en algún proveedor
\textit{cloud} se puede definir una propiedad y así dejar de utilizar Ollama local y usar una API de pago.

Para interactuar con el LLM desde código Java se utiliza el \textit{framework} Spring AI\@.
Esta biblioteca se encarga de pedirle al modelo que la salida cumpla con un \textit{schema} para lograr la salida
esperada en formato JSON\@.
Además permite el reemplazo de un modelo LLM por otro, incluso de distintos fabricantes, sin necesidad de realizar
cambios en la implementación.

El \textit{prompt} cuenta con tres secciones, el \textit{prompt} de sistema (\textit{system prompt}), el \textit{prompt}
del usuario y el \textit{prompt} para que la respuesta tenga la estructura deseada.

\subparagraph{\textit{system prompt}}

El \textit{prompt} de sistema es fijo para todas las recomendaciones de todos los usuarios.
En la implementación se utilizó lo siguiente (Ver \autoref{lst:system-prompt-txt}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={\textit{System prompt} para que el LLM se comporte como un asistente de viajes.},
    label={lst:system-prompt-txt}
]
You are a travel itinerary assistant.
Keep descriptions short and practical.
Prefer famous landmarks, neighborhoods, museums, parks, markets, and viewpoints.
Use canonical place names that can be geocoded later.
\end{lstlisting}

Como se puede apreciar se establecen instrucciones generales para guiar al modelo en el estilo de respuesta, todavía
sin mencionar la estructura.

\subparagraph{\textit{user prompt}}

El \textit{prompt} del usuario contendrá la instrucción concreta y será parametrizado con los intereses del usuario,
la ciudad deseada y la cantidad de días elegidos.
Para todas las recomendaciones se decidió fijar el valor fijo de tres puntos de interés por día.

Un ejemplo de \textit{prompt} de usuario para armar un itinerario de 3 días en la ciudad de Montreal, Canadá es el
siguiente (Ver \autoref{lst:user-prompt-txt}):

\begin{lstlisting}[
    style=basecodestyle,
    caption={\textit{User prompt} para que el LLM responda con recomendaciones para armar un itinerario concreto.},
    label={lst:user-prompt-txt}
]
Build a sightseeing itinerary for Montreal, Canada for 3 days.

User interests: Culture & Historic, Nature

Requirements:
- Return exactly 3 days in the itinerary.
- Each day must include:
  - day: the day number starting at 1
  - title: a short title
  - summary: a one-sentence summary
  - places: exactly 3 places to visit for that day
- Each place must include:
  - name
  - description: one concise sentence
  - category: one of CULTURE_HISTORIC, NATURE, SHOPPING, NIGHTLIFE, RELIGIOUS, NONE
- Avoid temporary events or unstable recommendations.
- The city field must be "Montreal".
- The country field must be "Canada".
- The days field must be 3.
- Every single day must have exactly 3 places.
- Favor places aligned with the user interests when possible.
- If the user interests are not provided or do not clearly apply, fall back to the city's general highlights.
- You only know these allowed category values: CULTURE_HISTORIC, NATURE, SHOPPING, NIGHTLIFE, RELIGIOUS, NONE
- Do not use any other category values beyond CULTURE_HISTORIC, NATURE, SHOPPING, NIGHTLIFE, RELIGIOUS, NONE.
- Set category to the best-fitting general category for the POI, even if it is not one of the user's preferred interests.
- If a POI does not clearly fit any category, set category to NONE.
\end{lstlisting}

En primer lugar se indica la instrucción concreta indicando la ciudad, el país de la ciudad y la cantidad de días.
A continuación se enumeran los intereses del usuario (en este caso \texttt{CULTURE\_HISTORIC} y \texttt{NATURE}).

Luego se menciona una serie de requerimientos y recomendaciones que se fueron obteniendo luego de varias pruebas
interactuando con el modelo.
Como se puede apreciar en el \textit{prompt} se pide organizar las recomendaciones por día, donde para cada día se
indica el número de día, un título y una breve descripción.
Cada día contará con exactamente tres puntos de interés.
Cada punto de interés debe contar con el nombre, una oración que lo describa y si se ajusta a una de las cinco
categorías establecidas.

A continuación todas las demás oraciones son instrucciones para que el modelo no cometa errores.
Si bien varias de ellas parecen redundantes, se fueron agregando luego de varias pruebas, siempre con la intención de
reforzar la instrucción y obtener el resultado esperado en el primer intento.
Allí se puede ver la instrucción ``Favor places aligned with the user interests when possible.'' para que priorice los% chktex 38
puntos de interés que el usuario espera.

Por último, Spring AI completa el \textit{prompt} agregando una tercera sección luego del \textit{system prompt} y el
\textit{user prompt}.
Aquí el \textit{framework} le exige al modelo que la respuesta debe ser estructurada y le provee el \textit{schema}

\subsubsection{Geocodificación de los puntos de interés}

Para poder mostrar las recomendaciones obtenidas en un mapa es necesario contar con las coordenadas geográficas
(latitud y longitud) de cada punto de interés.

Inicialmente se le solicitaba al LLM para que conteste también con las coordenadas de cada punto recomendado
pero los resultados resultaron siempre incorrectos.
Un LLM no es una fuente confiable de datos geoespaciales precisos ya que las respuestas que genera son siempre
probabilísticas y puede ``alucinar'' coordenadas inexistentes.
Por ejemplo en las pruebas para la ciudad de Toronto, todas las coordenadas que respondía el LLM estaban
desplazadas varios kilómetros y no sirven para lo que se espera en esta implementación.

Es por eso que se decidió utilizar un servicio de geocodificación para que, a partir del nombre del punto de interés, la
ciudad y el país se obtengan las coordenadas geográficas precisas de ese punto.
La API de~\cite{nominatim-api-overview} ofrece un uso gratuito con
la limitación de geolocalizar una cadena de texto por segundo.
En la implementación se tienen en cuenta estas restricciones por lo que el proceso de localizar todos los puntos de
interés de un itinerario puede demorar varios segundos.
Para eliminar esta restricción se puede utilizar un proveedor pago o el despliegue de una instancia propia.

Para cada geocodificación se realiza un \textit{request} HTTP a la API REST de Nominatim usando el path
\texttt{/search} donde el texto de consulta o \textit{query} es siempre el nombre del punto de interés, seguido del
nombre de la ciudad y luego del nombre del país, separado por coma.
Por ejemplo para el punto de interés ``CN Tower'' de la ciudad de Toronto en Canadá la \textit{query} es la cadena
``CN Tower, Toronto, Canada''.
En el \textit{request} se indica además que el formato de respuesta debe ser \texttt{jsonv2} y que la cantidad de
resultados es una.
Así la respuesta consiste en un JSON con tres campos relevantes: \texttt{lat}, \texttt{lon} y \texttt{display\_name}
(que corresponde al nombre completo y en esta implementación no es utilizado).

Un ejemplo de respuesta para la geocodificación del punto de interés ``Royal Ontario Museum, Toronto, Canada'' es el
siguiente (Ver \autoref{lst:api-nominatim-response-json}):

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Ejemplo de respuesta de la API de Nominatim.},
    label={lst:api-nominatim-response-json}
]
{
  "lat": 43.6676565,
  "lon": -79.3946988,
  "display_name": "Royal Ontario Museum, 100, Queen's Park, University, University-Rosedale, Toronto, Golden Horseshoe, Ontario, M5S 2C6, Canada"
}
\end{lstlisting}

Para algunas recomendaciones del LLM, dado que no coincide con el nombre exacto que el servicio de
geocodificación espera, no se podrán conseguir las coordenadas geográficas.
En la implementación se decidió utilizar las coordenadas del aeropuerto para marcar al menos la zona de la ciudad donde
el punto de interés se encuentra.
