Luego al presionar ``New Trip'' podrá arrastrar al navegador un archivo .eml con el correo electrónico sobre el

El usuario deberá exportar el correo en formato .eml usando la opción exportar de un cliente de escritorio de correo

Elegido el archivo basta con presionar ``Create Trip'' para que se dispare el \textit{pipeline} de extracción.

Todos los correos electrónicos procesados con éxito son asociados al usuario logueado y persistidos en la base de datos

Allí se listan todos los viajes creados a partir de la extracción, donde para cada correo electrónico se genera un

Además se ofrece el filtro por año de viaje.
Para cada viaje se indica el destino final (la ciudad correspondiente al aeropuerto de destino del último segmento),
el país del destino final, un nombre autogenerado (por ejemplo ``Trip to Montréal'') que el usuario puede modificar,
las fechas de inicio y fin del viaje, la cantidad de segmentos que tiene el viaje y la cantidad de itinerarios de POIs

En la página principal, además de la sección ``My Trips'', se cuenta con la sección ``My Interests''.

Si el usuario lo desea puede presionar ``Save'' y así persistir el itinerario completo con las recomendaciones obtenidas
para consultarlo más adelante.
La implementación permite almacenar un itinerario de recomendaciones por ciudad destino por viaje.

\subsubsection{Persistencia: Base de Datos}

Se requiere una base de datos relacional para poder almacenar los datos del usuario y cada uno de sus viajes, incluyendo
los resultados de todas las extracciones y los puntos de interés de todas las recomendaciones generadas.
Se decidió utilizar PostgreSQL porque es la base de datos relacional más popular.

Respecto al manejo de usuarios registrados en la aplicación se modela la tabla \texttt{user\_account} donde, para cada
usuario, se almacena la dirección de correo electrónico (que actúa al mismo tiempo como nombre de usuario) y un
\textit{hash} de la contraseña del usuario.

Cada usuario cuenta con uno o más viajes, modelados por la tabla \texttt{trip}.
Esta tabla cuenta con el nombre del viaje, el nombre del pasajero, el código de reserva, las fechas de inicio y fin
del viaje y los nombres de la ciudad y país destino del viaje.

Cabe mencionar que, para facilitar la implementación, existen campos en las tablas de la base que podrían ser obtenidos
con una invocación al servicio de \textit{enrichment}.
Sin embargo se opta por la persistencia de esta información para reducir las invocaciones al servicio indicado.
Además se facilita la implementación dado que la estructura de la tabla coincide con el archivo ``JSON'' que se obtiene
a la salida del \textit{pipeline} de extracción.

Cada viaje cuenta con uno o más segmentos de vuelo, modelados por la tabla \texttt{trip\_segment}.
Cada segmento cuenta con el número de vuelo y toda la información relevante tanto para el origen como para el destino
del segmento.
Esto incluye fecha y hora, código y nombre del aeropuerto, ciudad y país del aeropuerto, latitud y longitud del
aeropuerto y links útiles del aeropuerto (como el sitio web del mismo o el enlace a una entrada de Wikipedia sobre el
aeropuerto).

Por último es necesario almacenar las recomendaciones de POIs.
Para cada recomendación se persiste una tupla en la tabla \texttt{trip\_saved\_recommendation} donde se indica la
ciudad, el país y la cantidad de días elegida.
Para cada día de una recomendación existe un registro en la tabla \texttt{trip\_saved\_recommendation\_day}, incluyendo
dos campos de texto con título y el resumen de ese día.
Finalmente, para cada punto de interés de un día de una recomendación, la tabla
\texttt{trip\_saved\_recommendation\_place} lista el nombre, la descripción del POI, la categoría (según los cinco
intereses posibles ya mencionados) y las coordenadas geográficas (latitud y longitud) del mismo.

\paragraph{Flyway}

Para el modelado de la base de datos se utilizó Flyway, una herramienta que permite definir scripts SQL incrementales,
de forma que, al iniciar la aplicación, se ejecutan automáticamente los scripts que falten.
Esto facilita el agregado de nuevas tablas y campos y permite el versionado del \textit{schema} de la base de datos.

Por ejemplo, el primer \textit{script}, llamado \nolinkurl{V1\_\_create\_user\_account.sql} contiene únicamente la
definición de la tabla que modela a los usuarios (Ver \autoref{lst:flyway-1}) ya que es lo único que se necesitaba
para realizar una primera implementación de la funcionalidad de registro y login.

\begin{lstlisting}[
    style=sqlcodestyle,
    caption={Script de Flyway con la creación de la tabla de usuarios.},
    label={lst:flyway-1}
]
CREATE TABLE user_account (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    CONSTRAINT uk_user_account_email UNIQUE (email)
);
\end{lstlisting}

En la etapa final de la implementación se necesitó agregar los campos latitud y longitud para los POIs de las
recomendaciones.
Para ello se creó el script \nolinkurl{V5\_\_add\_recommendation\_place\_coordinates.sql} (Ver \autoref{lst:flyway-5})
con la sentencia \texttt{ALTER TABLE} correspondiente.

\begin{lstlisting}[
    style=sqlcodestyle,
    caption={Script de Flyway con el agregado de coordenadas de POIs},
    label={lst:flyway-5}
]
ALTER TABLE trip_saved_recommendation_place
    ADD COLUMN latitude DOUBLE PRECISION,
    ADD COLUMN longitude DOUBLE PRECISION;
\end{lstlisting}

\subsubsection{Back-End: Spring Boot}

Para la implementación del \textit{back-end} se decidió utilizar Spring Boot porque facilita la implementación de
aplicaciones web escritas en el lenguaje Java.
Spring Boot se destaca por reducir la configuración necesaria para conseguir una aplicación web desplegada gracias a
dependencias por defecto, convenciones establecidas y la autoconfiguración.
Permite integrar de forma sencilla varios componentes necesarios como APIs REST, acceso a base de datos, seguridad,
entre otras.

El \textit{back-end} se separó en seis módulos distintos, utilizando la herramienta Maven para organizar proyectos Java.
A continuación el detalle de los módulos de \textit{back-end}, donde los tres primeros ya fueron explicados en detalle
en secciones anteriores:

\begin{itemize}
    \item \texttt{extractor-core}: Contiene todo lo relacionado al \textit{pipeline} de extracción de Stanford NLP
    \item \texttt{extractor-enricher}: Enriquecimiento de la información
    \item \texttt{extractor-recommender}: Recomendación de puntos de interés
    \item \texttt{extractor-server}: Implementación del \textit{back-end} que invoca los tres módulos anteriores,
administra el acceso de los usuarios y logra la persistencia en la base de datos.
    \item \texttt{extractor-api}: Conjunto de clases comunes al proyecto (definiciones compartidas por más de un módulo)
    \item \texttt{extractor-common}: Métodos utilitarios utilizados por más de un módulo (como la lectura de un CSV)
\end{itemize}

\paragraph{Spring Data}

Se utilizó Spring Data para simplificar la comunicación entre el \textit{back-end} y la base de datos.
Permite abstraer gran parte del código fuente necesario para la interacción con la base mediante la definición de
repositorios con interfaces, de forma que el \textit{framework} genera automáticamente el comportamiento necesario.

Por ejemplo, se definió un \texttt{TripRepository} (Ver \autoref{lst:spring-data-trip-repository}) con sólo dos
operaciones, de forma que Spring Data genera automáticamente las \textit{queries} a la base, sólo analizando el nombre
que se le dio a los métodos (que siguen una convención ya definida por la herramienta).

\begin{lstlisting}[
    style=javacodestyle,
    caption={Repositorio de Viajes usando Spring Data},
    label={lst:spring-data-trip-repository}
]
public interface TripRepository extends JpaRepository<Trip, Long> {

    List<Trip> findAllByUserEmailOrderByIdDesc(String email);

    Optional<Trip> findByIdAndUserEmail(Long id, String email);

}
\end{lstlisting}

Así no se necesitan implementar manualmente \textit{queries} de \texttt{SELECT} para conseguir todos los viajes de
un usuario, basta con invocar a \texttt{findAllByUserEmailOrderByIdDesc} dado que se indica que el tipo de dato de
retorno es una lista de viajes (\texttt{List<Trip>}) y sólo se necesita el identificador del usuario (el email).

\paragraph{Spring Security}

Otra herramienta de la familia Spring es Spring Security, que apunta a gestionar de una forma sencilla la seguridad
de las aplicaciones en cuanto a la autenticación y autorización.
Su función principal es controlar quién puede acceder al sistema y qué acciones puede realizar el usuario una vez
logueado.

Para la implementación basta con definir un método donde se indiquen los \textit{endpoints} que requieren autenticación
y cuáles no (Ver \autoref{lst:spring-security-filter-chain}). Por ejemplo se indica que todos pueden acceder a
\texttt{/login} y \texttt{/register} de forma que todo lo que no está mencionado se restringe automáticamente (como
por ejemplo el endpoint de \texttt{core-enricher}).

\begin{lstlisting}[
    style=javacodestyle,
    caption={Lógica de autenticación usando Spring Security},
    label={lst:spring-security-filter-chain}
]
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
            .authorizeHttpRequests(authorize -> authorize
                    .requestMatchers("/", "/login", "/register", "/error").permitAll()
                    .requestMatchers("/api/auth/register").permitAll()
                    .requestMatchers("/assets/**").permitAll()
                    .anyRequest().authenticated()
            )
            .csrf(CsrfConfigurer::spa)
            .formLogin(form -> form
                    .loginPage("/login")
                    ...
                    .permitAll()
            )
            ...
}
\end{lstlisting}

Como se ve en el fragmento de código, se utilizó una protección contra solicitudes maliciosas basada en tokens
Cross-Site Request Forgery (CSRF) que consisten en valores únicos generados por el servidor y verificados en cada
solicitud que modifique el estado de la aplicación.
De esta forma, en caso de que un atacante envíe una petición maliciosa, al no tener el token esperado la operación será
rechazada.

\subsubsection{Front-End: Vanilla JS}

La interfaz de usuario requerida para la implementación cuenta con solo cuatro vistas: registro, login, listado de
viajes y detalle de un viaje.
Dada la simpleza del requerimiento se decidió utilizar JavaScript nativo (Vanilla JS) para conseguir una aplicación
web, en principio pensada para \textit{desktop} pero con funcionalidad \textit{responsive} para poder ser utilizada
también desde dispositivos móviles con pantallas más pequeñas.

En el directorio \texttt{assets/} del módulo \texttt{extractor-client} se ubican los archivos JavaScript y CSS
necesarios.
Además se utilizó Bootstrap, un framework front-end de componentes y reglas CSS que permite construir aplicaciones web
\texttt{responsive} consistentes a partir de estilos y componentes ya definidos.

Los archivos de extensión HTML se encuentran ubicados en el directorio \texttt{templates/} del módulo
\texttt{extractor-server}.
Esto es así porque se decidió utilizar Thymeleaf, un motor de plantillas para Java que permite generar HTML de forma
dinámica, insertando así datos provenientes del servidor dentro de plantillas HTML de forma sencilla que luego se
renderizan en la página.

Por último, y como ya se mencionó en \autoref{subsec:architecture} desde el \textit{front-end} se invoca a la API
de MediaWiki para conseguir una imagen de la ciudad que corresponde al destino final del viaje.
Esta imagen se puede ver en la sección ``My Trips''.
La API es gratuita y para esta implementación se realiza un \texttt{request} GET (Ver \autoref{lst:js-mediawiki})
probando primero con la cadena \texttt{Ciudad, País}.
En caso de obtener un resultado se intenta luego con las cadenas \texttt{Ciudad (País)} y \texttt{Ciudad}.
La respuesta de la API es un JSON que contiene la URL directa del recurso buscado, esto es, la imagen representativa
de una ciudad.
Para reducir las invocaciones a la API se implementa una caché a nivel de la sesión del lado del navegador.

\begin{lstlisting}[
    style=javacodestyle,
    caption={Uso de la API de MediaWiki para imagen de una ciudad},
    label={lst:js-mediawiki}
]
const fetchImageUrlForTitle = async (title) => {
    const url = new URL("https://en.wikipedia.org/w/api.php");
    url.searchParams.set("action", "query");
    url.searchParams.set("format", "json");
    url.searchParams.set("origin", "*");
    url.searchParams.set("redirects", "1");
    url.searchParams.set("prop", "pageimages");
    url.searchParams.set("piprop", "thumbnail");
    url.searchParams.set("pithumbsize", "640");
    url.searchParams.set("titles", title);

    const response = await fetch(url);
};
\end{lstlisting}

\subsubsection{Docker}

Para facilitar el despliegue de la aplicación se utiliza Docker.
Para ello se define un archivo Dockerfile (Ver \autoref{lst:docker-dockerfile}) en la raíz del proyecto
\texttt{extractor-parent} que se encarga de crear
la imagen con todos los módulos del proyecto en dos etapas, lo que se conoce como \textit{multi-stage build}.
En la primera etapa se compila la aplicación por completo, a partir de la imagen base \texttt{eclipse-temurin:25-jdk}.
En lugar de ejecutar la aplicación sobre la misma imagen utilizada para la compilación, se provee solo el .jar
resultante a una imagen más liviana, a partir de la imagen base \texttt{eclipse-temurin:25-jre}, reduciendo así el
tamaño final del contenedor.
Por último se inicia la aplicación mediante el comando \texttt{java -jar app.jar}.

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Archivo Dockerfile (se omiten varias líneas (...) correspondientes a la etapa \texttt{build})},
    label={lst:docker-dockerfile}
]
FROM eclipse-temurin:25-jdk AS build

WORKDIR /workspace

(...)

RUN --mount=type=cache,target=/root/.m2 \
    ./mvnw -pl server -am package -Dmaven.test.skip=true

(...)

FROM eclipse-temurin:25-jre

WORKDIR /app

COPY --from=build /workspace/server/target/extractor-server-1.0-SNAPSHOT.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java", "-jar", "/app/app.jar"]

\end{lstlisting}

En el archivo de Docker Compose \texttt{compose.yaml} (Ver \autoref{lst:docker-compose}), también ubicado en la raíz del
proyecto, se define la orquestación completa del entorno de ejecución de la aplicación.
Docker Compose es una herramienta que permite definir cómo se deben iniciar contenedores de Docker utilizando un
único archivo YAML\@.

En primer lugar se puede ver cómo se indica que se debe instanciar un contenedor con la base de datos, esto es,
utilizando una imagen de Postgres 18, en el servicio \texttt{postgres}.
En \texttt{environment} se indican las propiedades que deben ser reemplazadas en ejecución.
Estos valores están definidos en un archivo .env para centralizar las variables de configuración del entorno,
evitando dejarlas definidas en el código y permitiendo contar con más de un archivo .env para realizar el despliegue
de la aplicación en distintos ambientes (por ejemplo un ambiente local y otro en la nube).

Luego el servicio \texttt{ollama} se define a partir de la imagen Ollama 0.20.
Para la descarga a demanda del modelo LLM requerido se define otro servicio \texttt{ollama-init} que descarga el modelo
y luego finaliza su ejecución.

Por último el servicio \texttt{webapp} es el que sirve la aplicación web, tanto el \texttt{back-end} como el
\texttt{front-end}.
Como se puede apreciar, para que este servicio funcione, necesita que se le indiquen las credenciales de la base de
datos, la URL del servicio de Ollama, y la configuración de la API de Nominatim.

En todos los servicios se definen los puertos correspondientes en la sección \texttt{ports} y el volumen compartido
en la sección \texttt{volumes}.
Se agregaron verificaciones de \texttt{healthcheck} y \texttt{depends\_on} para que los servicios se inicien en el orden
correspondiente.
Esto evita por ejemplo que la aplicación web inicie antes de que la base de datos esté lista o para obtener un mensaje
cuando alguno de los contenedores finaliza su ejecución de forma inesperada.

\begin{lstlisting}[
    style=jsonlikecodestyle,
    caption={Archivo compose.yaml (se omiten varias líneas (...))},
    label={lst:docker-compose}
]
services:
  postgres:
    image: postgres:18-alpine
    (...)
    ports:
      - "${POSTGRES_PORT}:5432"
    volumes:
      - postgres:/var/lib/postgresql
  ollama:
    image: ollama/ollama:0.20.3
    (...)
    ports:
      - "${OLLAMA_PORT}:${OLLAMA_PORT}"
    volumes:
      - ollama:/root/.ollama
  ollama-init:
    image: ollama/ollama:0.20.3
    (...)
    entrypoint:
      - /bin/sh
      - -c
      - |
        ollama serve &
        server_pid=$$!
        until ollama list >/dev/null 2>&1; do
          sleep 1
        done
        ollama pull ${OLLAMA_MODEL}
        kill $$server_pid
        wait $$server_pid || true
        sleep "${OLLAMA_BOOTSTRAP_COOLDOWN_SECONDS}"
    volumes:
      - ollama:/root/.ollama
  webapp:
    image: extractor-webapp:latest
    build:
      context: .
    ports:
      - "${APP_PORT}:${APP_PORT}"
    volumes:
      - ./logs:/app/logs

volumes:
  ollama:
  postgres:
\end{lstlisting}
